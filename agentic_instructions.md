# Agentic Instructions

## Purpose

This document defines how AI tools should assist in the development of the **Healthcare Auto Adjudication Report Automation Toolkit**.

AI is used as a productivity accelerator for scaffolding, documentation, and repetitive coding tasks.
Core system logic, workflow design, validation rules, and architecture decisions remain the responsibility of the developer.

The system must reflect real-world healthcare reporting workflows rather than generic automation patterns.

---

## Development Philosophy

1. AI assists, developer decides.
2. Business logic must be written or validated by the developer.
3. AI-generated code must always be reviewed before committing.
4. Prefer simple, readable solutions over complex frameworks.
5. Automate repetitive work but keep the system understandable.

---

## Product Goal

Build a lightweight automation system that processes healthcare auto-adjudication reports generated by legacy systems and automatically delivers clean reports to SharePoint.

The product replaces a manual process:

Mainframe job → Outlook email → Excel cleaning → SharePoint upload → Daily reporting

with an automated pipeline.

---

## Core Principles

**Developer-owned logic**

The following must be written or designed primarily by the developer:

* Data cleaning rules
* Validation rules
* Report schema interpretation
* Business-specific field transformations
* Failure handling logic

These represent domain knowledge and should not rely blindly on AI suggestions.

---

**AI-assisted tasks**

AI tools may assist with:

* Project scaffolding
* Boilerplate API code
* Docker setup
* CI/CD configuration
* Logging utilities
* Test generation
* Documentation
* Code refactoring

AI suggestions must always be verified.

---

## System Overview

Input
TXT or CSV reports generated by legacy systems (mainframe batch jobs).

Processing
Python pipeline cleans, validates, and transforms the data.

Output
Clean report automatically uploaded to SharePoint.

Additional components:

* Daily summary email
* Lightweight monitoring dashboard

## AI Collaboration Rules

### 1. Code Generation

AI may generate:

* helper functions
* file parsers
* API endpoints
* test cases
* documentation

AI should **not invent domain rules** unless explicitly provided by the developer.

---

### 2. Data Processing Logic

Developer defines:

* required columns
* validation rules
* missing data handling
* formatting standards
* report structure

AI may assist with Pandas implementations but must follow developer specifications.

---

### 3. Refactoring

AI may suggest improvements to:

* readability
* modularization
* error handling
* logging

Refactoring should not alter business logic.

---

### 4. Testing

AI can generate:

* unit tests
* edge case tests
* mock datasets

Developer must verify test coverage reflects real reporting scenarios.

## Development Workflow

### Step 1

Define report schema.

Understand the structure of the TXT or CSV reports generated by the legacy system.

Identify:

* columns
* field formats
* null values
* identifiers
* timestamps

---

### Step 2

Build ingestion module.

Responsibilities:

* parse incoming files
* normalize file format
* detect encoding issues
* store raw data safely

---

### Step 3

Build processing pipeline.

Core responsibilities:

* cleaning data
* validating fields
* transforming formats
* producing standardized output

Most domain logic lives here.

---

### Step 4

Implement output module.

Responsibilities:

* upload processed files to SharePoint
* handle authentication
* verify successful upload

---

### Step 5

Add notification layer.

Daily summary email should include:

* number of reports processed
* number of validation failures
* processing time
* error messages if any

---

### Step 6

Expose simple API.

FastAPI endpoints may include:

* trigger pipeline
* check processing status
* fetch report metrics

---

### Step 7

Add dashboard.

Dashboard should visualize:

* daily processing counts
* success vs failure
* processing time trends

Keep it lightweight.

---

## CI/CD Expectations

GitHub Actions should automate:

* linting
* testing
* Docker image build
* deployment validation

Avoid complex pipelines early.

---

## Deployment Philosophy

This tool is designed to run as:

* scheduled container job
* internal reporting utility
* automation service

Docker ensures portability across environments.

---

## Guardrails

AI-generated code must never:

* introduce unnecessary frameworks
* hide important logic inside abstractions
* reduce transparency of the pipeline
* replace developer-defined validation logic

Simplicity is preferred over cleverness.

---

## Definition of Done

A feature is considered complete when:

* code is readable
* logic is tested
* documentation is updated
* pipeline works end to end
* manual workflow is reduced

---

## Long Term Vision

Possible future improvements:

* automated Outlook ingestion
* anomaly detection in reports
* historical analytics
* scheduled orchestration
* multi-report pipeline support

The system should remain simple, transparent, and maintainable.
